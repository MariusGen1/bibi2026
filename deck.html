Bibi 02 — Customer & sub-job
Who we're solving for

Luigi runs a
25-table trattoria.

· 28 menu items, 14 years in business
· ~300 covers / week
· Two cooks on the line, three servers, one host
· No data analyst. No CRM. No time.
The sub-job
“Which items should I add, remove, or change?” — every menu rewrite, twice a year

A single, recurring, high-stakes decision — made today on intuition, comp tabs, and one loud regular.

Bibi 03 — Problem statement
What Luigi tries today

Every existing tool answers the wrong question.

01.
POS reports
Tell him what sold — not what was loved. A dish can sell and still leave half the plate.
02.
Yelp / Google reviews
Generic, public, weeks late. The furious and ecstatic write — the merely-happy don't.
03.
Paper comment cards
Almost no one fills them out. The ones who do mostly write “great service!”
04.
Asking servers
They relay whichever table was loudest. Always the same two people.

The missing thing: item-level signal, from the people who already love the place, delivered privately, in plain English.

Bibi 04 — What Bibi does
Three surfaces, one loop

Receipt → ratings → Monday-morning brief.

01.
QR on the receipt

Guest scans on the way out. Picks the dishes they ordered. Stars each one. Drops a comment if they feel like it.

02.
SMS discount

One-time code by text for their next visit. Captures the phone. Closes the loop. Average completion: 38 seconds.

03.
AI weekly brief

A short written summary that names dishes, not feelings. Three to fix, two to push. Refreshed every Monday.

Bibi 07 — Stack & architecture
What it's built on

One Next.js app, three surfaces.

Stack
Next.js 16 (App Router · Turbopack) React 19.2 Tailwind v4 TypeScript Supabase (Postgres) Twilio Anthropic Claude Opus 4.7 Vercel
Architecture in one breath

Server Actions write through the Supabase service role; the anon client is unused. AI summary is a single Claude call with ephemeral prompt caching on the system prompt. SMS goes through Twilio with a mock fallback so dev never blocks.

Three surfaces, one DB
/
Landing. Editorial marketing page.
/r/[slug]
Customer flow. Mobile-first ratings + SMS code.
/owner
Owner dashboard. Item analytics + AI brief.
Design system

Fraunces serif + Instrument Sans, cream paper, hairline rules, tomato accents, hand-drawn SVG icons. The product reads as a printed trattoria menu, not a SaaS dashboard.

Bibi 08 — Decisions & challenges
Key implementation decisions
01.
Server-only writes, no RLS.
Single-tenant demo. Service-role through server actions; anon client unused. Auth = HMAC-signed passcode cookie.
02.
AI brief on-demand, not cached.
Owner clicks “write this week's summary.” Cheap enough at hackathon scale; cache later.
03.
Mock SMS fallback.
If Twilio env vars are missing, log + on-screen preview. Dev path never blocks.
04.
Editorial design vocabulary.
Hairline rules instead of card shadows. Fraunces variable font. No gradients.
Technical challenges
01.
Next.js 16 + Tailwind v4 are new.
Training data caps at 15/v3. Read vendored docs in node_modules/next/dist/docs instead of guessing.
02.
CSS cascade mystery.
Unlayered button { color: inherit } beat Tailwind's .text-paper. Wrapped reset in @layer base.
03.
OWASP self-audit + hardening.
HMAC cookie, timing-safe compare, crypto.randomInt, IP rate limit, prompt-injection sandbox, HSTS + nosniff headers.
04.
Favicon, three takes.
BMP-in-ICO → PNG-in-ICO → RGBA PNG-in-ICO. Tiny, but it's the first pixel of the brand.

Bibi 10 — Building with AI · cons & wishlist
Cons
·
Stale framework knowledge.
Confidently wrote Next.js 15 patterns into a Next.js 16 codebase. Had to be explicitly pointed at vendored docs.
·
No browser eyes.
Visual issues (the black-on-black button, the muddy favicon) needed human screenshots to even notice.
·
Over-eager defaults.
Adds error handling, fallbacks, and abstractions we didn't ask for. Hard to keep the codebase lean without active pushback.
What we'd change
→
Read framework docs by default
on any major-version-bump dependency, instead of waiting to be told.
→
Built-in browser preview.
Let the model see the rendered page, not just the source.
→
Cheaper image-asset iteration.
Three favicon rebuilds wasted a real chunk of time. A “preview an asset at 16/32/48” primitive would have caught the muddiness in one round.
→
Less reflexive scaffolding.
Default to the minimum the task needs. Abstract on the third copy, not the first.
